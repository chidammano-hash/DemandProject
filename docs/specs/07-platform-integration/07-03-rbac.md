# 07-03 Role-Based Access Control (RBAC)

## Overview

JWT-based authentication and role-based authorization system. Users authenticate via login endpoint, receive a JWT token, and access is controlled by role-permission mappings. All mutations are audit-logged.

## Components

| Component | Path | Purpose |
|---|---|---|
| Auth module | `common/auth.py` | JWT issue/verify, bcrypt password hashing, RBAC permission checks |
| Auth router | `api/routers/auth_router.py` | Login, refresh, logout endpoints |
| Users router | `api/routers/users.py` | User CRUD, role assignment, profile endpoints |
| DDL | `sql/063_create_users_rbac.sql` | `dim_user`, `dim_role`, `fact_audit_log` |
| Config | `config/auth_config.yaml` | JWT settings, role definitions, permission matrix |

## Auth Module (`common/auth.py`)

- `hash_password(plain: str) -> str` — bcrypt hash
- `verify_password(plain: str, hashed: str) -> bool` — bcrypt verify
- `create_access_token(user_id: int, role: str, expires_minutes: int) -> str` — JWT with `sub`, `role`, `exp`
- `create_refresh_token(user_id: int) -> str` — longer-lived JWT for token refresh
- `decode_token(token: str) -> dict` — verify + decode, raises `HTTPException(401)` on expiry/invalid
- `require_role(*roles: str)` — FastAPI dependency; extracts token from `Authorization: Bearer`, checks role membership
- `require_permission(permission: str)` — FastAPI dependency; checks role-permission matrix from config

## API Endpoints

### Auth Router (`/auth`)

| Method | Path | Description |
|---|---|---|
| POST | `/auth/login` | Authenticate with email + password, returns access + refresh tokens |
| POST | `/auth/refresh` | Exchange refresh token for new access token |
| POST | `/auth/logout` | Invalidate refresh token (server-side blocklist) |

### Users Router (`/users`)

| Method | Path | Auth | Description |
|---|---|---|---|
| GET | `/users` | admin | List all users with roles |
| POST | `/users` | admin | Create user (email, name, role, password) |
| GET | `/users/me` | any | Current user profile from token |
| PUT | `/users/{id}` | admin | Update user fields |
| PUT | `/users/{id}/role` | admin | Change user role |
| DELETE | `/users/{id}` | admin | Deactivate user (soft delete) |

## Database Schema

### `dim_user`
- `user_id SERIAL PRIMARY KEY`
- `email TEXT UNIQUE NOT NULL`, `display_name TEXT NOT NULL`
- `password_hash TEXT NOT NULL`
- `role TEXT NOT NULL DEFAULT 'viewer'`
- `is_active BOOLEAN DEFAULT true`
- `created_at TIMESTAMPTZ DEFAULT NOW()`, `last_login_at TIMESTAMPTZ`

### `dim_role`
- `role_name TEXT PRIMARY KEY`
- `permissions TEXT[] NOT NULL` (array of permission strings)
- `description TEXT`

### `fact_audit_log`
- `log_id BIGSERIAL PRIMARY KEY`
- `user_id INTEGER REFERENCES dim_user(user_id)`
- `action TEXT NOT NULL` (create, update, delete, login, logout)
- `resource TEXT NOT NULL` (table or endpoint name)
- `resource_id TEXT`, `details JSONB`
- `ip_address INET`, `created_at TIMESTAMPTZ DEFAULT NOW()`
- Index: `(user_id, created_at DESC)`, `(resource, action)`

## Config (`config/auth_config.yaml`)

```yaml
jwt:
  secret_key_env: "JWT_SECRET_KEY"
  algorithm: "HS256"
  access_token_expires_minutes: 30
  refresh_token_expires_days: 7
roles:
  admin:
    permissions: ["*"]
  planner:
    permissions: ["read", "write:forecast", "write:policy", "write:exceptions", "run:jobs"]
  analyst:
    permissions: ["read", "run:jobs"]
  viewer:
    permissions: ["read"]
default_role: viewer
```

## Integration

- Existing `api/auth.py` (`require_api_key`) remains for backward compat; RBAC layer is additive
- Audit log middleware in `api/main.py` writes to `fact_audit_log` on every mutation
- JWT secret loaded from `JWT_SECRET_KEY` env var (required in production, auto-generated in dev)
