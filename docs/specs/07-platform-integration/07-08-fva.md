# 07-08 Forecast Value Added (FVA) & ROI Tracking

## Overview

Tracks the business value added by each stage of the forecasting process and by individual planner interventions. The FVA waterfall compares accuracy across model tiers (external baseline, champion selection, oracle ceiling), while the intervention tracker records specific planner actions (policy changes, safety stock adjustments, forecast overrides, AI insight actions) with before/after metrics and financial impact. Together they answer: "Is the planning process adding value, and where?"

## Architecture

```
Intervention recorded
       |
       v
fact_intervention_metrics (pending)
       |
       |-- GET /fva/waterfall      --> model-tier accuracy comparison
       |-- GET /fva/interventions   --> paginated intervention log
       |-- GET /fva/roi-summary     --> aggregate ROI metrics
       |
       v
measurement_window_end reached --> status: 'measured', actual_financial_impact populated
```

The waterfall is computed on-the-fly from `fact_external_forecast_monthly` by grouping WAPE-based accuracy by `model_id`. Interventions are tracked in `fact_intervention_metrics` with a configurable measurement window per intervention type; once the window expires and actuals are available, `actual_financial_impact` is populated and `status` flips from `pending` to `measured`.

## Database Schema

### `fact_intervention_metrics`

DDL: `sql/068_create_fva_tracking.sql`

| Column | Type | Description |
|---|---|---|
| `intervention_id` | `BIGSERIAL PRIMARY KEY` | Auto-increment ID |
| `user_id` | `UUID` | User who made the intervention (nullable) |
| `intervention_type` | `TEXT NOT NULL` | Type: `forecast_override`, `ss_change`, `policy_change`, `ai_insight_action`, `sop_approval` |
| `resource_type` | `TEXT` | Target entity type (e.g., `dfu`, `policy`, `plan`) |
| `resource_id` | `TEXT` | Target entity identifier (e.g., `100320-1401`) |
| `metric_before` | `JSONB` | Snapshot of key metrics before intervention |
| `metric_after` | `JSONB` | Snapshot of key metrics after measurement window |
| `financial_impact_estimate` | `NUMERIC` | Estimated financial impact at time of intervention ($) |
| `actual_financial_impact` | `NUMERIC` | Measured financial impact after window expires ($) |
| `measurement_window_start` | `DATE` | Start of the measurement period |
| `measurement_window_end` | `DATE` | End of the measurement period |
| `status` | `TEXT DEFAULT 'pending'` | Lifecycle: `pending` (within window) or `measured` (actuals available) |
| `created_at` | `TIMESTAMPTZ DEFAULT now()` | Record creation timestamp |

### Indexes

| Name | Column(s) | Purpose |
|---|---|---|
| `idx_intervention_user` | `user_id` | Filter interventions by planner |
| `idx_intervention_status` | `status` | Separate pending vs measured |
| `idx_intervention_type` | `intervention_type` | Filter by intervention category |
| `idx_intervention_window_end` | `measurement_window_end` | Identify interventions ready for measurement |

## API Endpoints

All endpoints are read-only (no auth required).

### `GET /fva/waterfall`

FVA waterfall comparing accuracy across model tiers.

**Query params:**
- `months` (int, default 12, range 1-36) -- lookback window

**Response:**
```json
{
  "months": 12,
  "waterfall": {
    "external": { "model_id": "external", "accuracy_pct": 72.5, "n_rows": 1000 },
    "champion": { "model_id": "champion", "accuracy_pct": 78.3, "n_rows": 1000 },
    "ceiling":  { "model_id": "ceiling",  "accuracy_pct": 85.1, "n_rows": 1000 },
    "models": [ ... ]
  }
}
```

**SQL:** Queries `fact_external_forecast_monthly` grouped by `model_id`, filtering to `lag = 0` rows with non-null actuals within the lookback window. Accuracy formula: `100 - (100 * SUM(|F-A|) / |SUM(A)|)`.

### `GET /fva/interventions`

Paginated list of tracked interventions with before/after metrics.

**Query params:**
- `intervention_type` (string, optional) -- filter by type
- `status` (string, optional) -- filter by status (`pending` or `measured`)
- `user_id` (string/UUID, optional) -- filter by planner
- `limit` (int, default 50, max 500) -- page size
- `offset` (int, default 0) -- page offset

**Response:**
```json
{
  "total": 5,
  "interventions": [
    {
      "intervention_id": 1,
      "user_id": null,
      "intervention_type": "policy_change",
      "resource_type": "dfu",
      "resource_id": "100320-1401",
      "metric_before": { ... },
      "metric_after": null,
      "financial_impact_estimate": 5000.0,
      "actual_financial_impact": null,
      "measurement_window_start": "2025-01-01",
      "measurement_window_end": "2025-06-30",
      "status": "pending",
      "created_at": "2025-03-01T12:00:00"
    }
  ]
}
```

### `GET /fva/roi-summary`

Aggregate ROI metrics across all interventions.

**Query params:**
- `months` (int, default 12, range 1-36) -- lookback window

**Response:**
```json
{
  "months": 12,
  "total_interventions": 10,
  "measured": 4,
  "pending": 6,
  "total_estimated_impact": 50000.0,
  "total_actual_impact": 20000.0
}
```

## Configuration

`config/fva_config.yaml`

```yaml
measurement_windows:
  forecast_override: 3    # months to wait before measuring outcome
  ss_change: 6
  policy_change: 6
  ai_insight_action: 3
  sop_approval: 6

financial_impact:
  carrying_cost_pct: 0.25           # annual carrying cost as % of inventory value
  stockout_cost_per_unit: 50        # estimated cost per unit stockout
  service_level_target: 0.95        # target fill rate for impact calculations
```

**Measurement windows** define how long to wait after an intervention before measuring its outcome. Short-horizon actions (forecast overrides, AI insight responses) use 3-month windows; structural changes (policy, safety stock, S&OP approvals) use 6-month windows.

**Financial impact parameters** are used to estimate the dollar impact of interventions: carrying cost for excess inventory, stockout cost for lost sales, and the service level target against which fill rate improvements are valued.

## Frontend (`FVATab.tsx`)

### Layout

1. **Header** -- "Forecast Value Added" title + month selector dropdown (3, 6, 12, 24 months)
2. **ROI KPI Cards** -- 4-card grid:
   - Total Interventions (count)
   - Measured (count of completed measurements)
   - Estimated Impact (dollar value, formatted as $XK)
   - Actual Impact (dollar value, formatted as $XK)
3. **Model Accuracy Waterfall** -- Recharts `BarChart` with per-model bars, color-coded:
   - External (slate `#94a3b8`)
   - Champion (blue `#3b82f6`)
   - Ceiling (emerald `#10b981`)
   - Planner (violet `#8b5cf6`)
4. **Recent Interventions** -- List of last 10 interventions, each showing:
   - Status dot (green = measured, amber = pending)
   - Intervention type + resource type/ID
   - Estimated financial impact
   - Creation date

### Query Integration

- `fvaKeys.waterfall(months)` / `fetchFVAWaterfall(months)` -- waterfall data
- `fvaKeys.roi(months)` / `fetchFVAROI(months)` -- ROI summary
- `fvaKeys.interventions` / `fetchFVAInterventions(limit)` -- intervention list
- All queries use `STALE_PLATFORM` (5 min) stale time

## Integration Points

Interventions are recorded by upstream features when planners take action:

- **Override Queue Panel** -- records `forecast_override` when a planner adjusts a DFU forecast
- **Policy Management Panel** -- records `policy_change` when replenishment policy is changed
- **Safety Stock Panel** -- records `ss_change` when safety stock targets are modified
- **AI Planner** -- records `ai_insight_action` when a planner accepts/resolves an AI insight
- **S&OP Tab** -- records `sop_approval` when a consensus plan is approved

A background job (or cron) periodically scans for interventions where `measurement_window_end <= now()` and `status = 'pending'`, computes `metric_after` and `actual_financial_impact` from current data, and updates `status` to `'measured'`.
