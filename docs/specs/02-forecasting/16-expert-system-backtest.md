# Expert System Backtest (ExpSys)

## 1. Purpose

Produces a canonical production forecast using a **segment-assigned algorithm** (the "Expert System"). Rather than running 26 algorithms and selecting via a portfolio optimizer, ExpSys directly assigns the empirically best algorithm per demand archetype, backtests it across the full DFU population, and loads the results to PostgreSQL.

**Goals:**
- Replace the multi-algorithm competition with a fast, deterministic single-assignment per archetype
- Record accuracy KPIs at each forecast lag (0–4) to quantify degradation over the horizon
- Load execution-lag predictions as the operational production forecast

---

## 2. Algorithm Assignment

Default assignment (configurable via `config/expert_system_backtest.yaml`):

| Archetype | Primary | Fallback chain |
|---|---|---|
| erratic_high | nbeats | chronos → mstl → rolling_mean |
| erratic_low | nbeats | chronos → mstl → rolling_mean |
| insufficient | chronos | mstl → rolling_mean |
| smooth_low | chronos | mstl → rolling_mean |
| smooth_high | mstl | chronos → rolling_mean |

Fallbacks activate per-DFU when the primary algorithm has insufficient history (e.g., nbeats requires 24 months, MSTL requires 25 months). `rolling_mean` always succeeds as the final fallback (requires ≥ 3 months).

---

## 3. Timeframe Structure and Lag Computation

10 expanding-window timeframes are generated by `common/ml/backtest_framework.py → generate_timeframes()`.

```
tf_idx  train_end     fcstdate      lag 0      lag 1   ... lag 4
  0     2025-03       2025-04       2025-04    2025-05     2025-08
  1     2025-04       2025-05       2025-05    2025-06     2025-09
  ...
  9     2025-12       2026-01       2026-01    2026-02     2026-05
```

**Key definitions:**
- `fcstdate` = `train_end + 1 month` — the month when the forecast is generated (= `predict_start`)
- `lag` = `months(startdate − fcstdate)` — forecast horizon from generation date
- `lag 0`: predicting the immediate next month after training (shortest horizon)
- `lag 4`: predicting 5 months ahead (longest horizon stored)
- Lag 5 is excluded — not stored in `backtest_lag_archive` (`CHECK lag BETWEEN 0 AND 4`)

For a calendar month M that falls within multiple timeframes' prediction windows, it accumulates lag-0 through lag-4 predictions from different training cutoffs. This enables measuring how accuracy degrades as the horizon grows.

---

## 4. Accuracy KPIs

Computed after all 10 timeframes are run, using inner-join with actual `fact_sales_monthly` data.

**Lag-level accuracy** (aggregated across all timeframes):

```
Lag 0   accuracy%   WAPE%   DFUs   DFU-months    ← shortest horizon
Lag 1   ...
...
Lag 4   accuracy%   WAPE%   DFUs   DFU-months    ← longest horizon
────────────────────────────────────────────────
ExecLag accuracy%   WAPE%   DFUs   DFU-months    ← each DFU at its own lag
```

**Per-segment breakdown** at each lag (erratic_high, erratic_low, insufficient, smooth_high, smooth_low).

Accuracy formula: `100 − (SUM(|F−A|) / |SUM(A)|) × 100`

## 5. Database Output

### 5a. `backtest_lag_archive` — all lags 0–4
`model_id = 'ExpSys'` (configurable via `algorithm_id` in config)

| Column | Source |
|---|---|
| `forecast_ck` | `{item_id}_{customer_group}_{loc}_{fcstdate:YYYYMMDD}` |
| `item_id`, `customer_group`, `loc` | from `dim_sku` |
| `fcstdate` | `train_end + 1 month` |
| `startdate` | the forecasted month |
| `lag` | `months(startdate − fcstdate)` |
| `execution_lag` | from `dim_sku` |
| `basefcst_pref` | forecast quantity |
| `tothist_dmd` | total historical sales for the DFU |
| `model_id` | `'ExpSys'` |
| `timeframe` | timeframe label (A–J) |

Upsert on `UNIQUE(forecast_ck, model_id, lag)`.

### 5b. `fact_production_forecast` — execution-lag only
`plan_version = 'ExpSys_v1'` (configurable)

Predictions are aggregated to **item + loc grain** (summed across customer groups) since the production forecast table does not include `customer_group`. Uses the most recent timeframe's prediction per (DFU, month). Upsert on `UNIQUE(plan_version, item_id, loc, forecast_month)`.

---

## 6. Full Population vs Sample

The expert panel uses `n_dfus: 5000` (stratified sample). This script loads **all DFUs** from `fact_sales_monthly` with no sampling. Typical population size: 40K–150K DFUs depending on data density.

---

## 7. Checkpointing

After each timeframe completes, predictions are saved to:
```
data/expert_system_backtest/tf_<label>_predictions.parquet
```

On restart, completed timeframes are loaded from disk and skipped. This allows resuming interrupted runs.

## 8. Configuration Reference

```yaml
# config/expert_system_backtest.yaml
experiment:
  n_timeframes: 10
  forecast_horizon: 6      # months predicted per timeframe
  max_lags: 4              # lags 0-4 stored (lag 5 excluded by DB constraint)
  algorithm_id: "ExpSys"   # model_id in backtest_lag_archive
  plan_version: "ExpSys_v1"
  output_dir: "data/expert_system_backtest"
  n_workers: 14            # parallel workers for mstl

assignment:
  erratic_high: nbeats
  erratic_low:  nbeats
  insufficient: chronos
  smooth_low:   chronos
  smooth_high:  mstl
  fallbacks:
    nbeats:   [chronos, mstl, rolling_mean]
    chronos:  [mstl, rolling_mean]
    mstl:     [chronos, rolling_mean]
    default:  [rolling_mean]

nbeats:
  variant: generic
  input_size: 24
  h: 6
  max_steps: 500
  batch_size: 32
  learning_rate: 0.001
  min_history: 24

chronos:
  model_size: small
  device: mps
  batch_size: 32
  prediction_length: 6
  num_samples: 20

mstl:
  season_length: 12
  min_history: 25

rolling_mean:
  window: 6
  min_history: 3
```

## 11. Expected Runtime

| Algorithm | DFUs (typical) | Per timeframe | 10 timeframes |
|---|---|---|---|
| nbeats (GPU) | ~9K erratic | ~15 min | ~2.5 h |
| chronos (GPU) | ~8K insuff+smooth_low | ~10 min | ~1.5 h |
| mstl (14 workers) | ~1K smooth_high | ~2 min | ~20 min |
| **Total** | | | **~4–5 h** |

Runtime depends heavily on DFU count, GPU availability, and model step counts. Use `--skip-load` to profile accuracy without DB overhead.
