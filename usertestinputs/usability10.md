## U10.1 — AI Planner FVA "Recent Runs" list dumps a raw 600-character Pydantic stack trace ("2 validation errors for Recommendation proposed_qty.1 Input should be a valid number [type=float_type…] visit https://errors.pydantic.dev/2.12/v/float_type") inline under a failed run row [P1] [usability]
- **Category:** usability
- **Evidence:** `aiPlannerFva.png` + digest 2724-2733 — the `failed / 2025-12-01 / ollama` run renders, directly beneath it, the verbatim backend exception: *"2 validation errors for Recommendation proposed_qty.1 Input should be a valid number [type=float_type, input_value=None, input_type=NoneType] For further information visit https://errors.pydantic.dev/2.12/v/float_type proposed_qty.2 …"*. Code: `AiPlannerFvaTab.tsx:291-300` renders `{r.error_message}` unmodified in a `colSpan={5}` row (also `:544` in the detail pane). Cycle-7 U7.2 fixed the *blank-panel* dead-end by surfacing `error_message`, but it surfaces the entire raw Pydantic `ValidationError.__str__()` — internal type names, `input_type=NoneType`, and a developer doc URL — to the planner.
- **Impact:** This is the single most unprofessional surface in the app: a planner skimming backtest history sees a wall of red framework-internal jargon and a `errors.pydantic.dev` link, with no plain-language reason ("recommendation parser rejected null quantities") and no next step. It also visually dominates the run list (the trace is taller than 4 run rows combined), pushing real runs below the fold.
- **File:** `frontend/src/tabs/AiPlannerFvaTab.tsx:291-300` (inline failed-row) + `:544` (detail pane). Ideally also `api/routers/...` FVA-backtest run handler should persist a sanitized `error_message` rather than `str(exc)`.
- **Recommendation:** Render a short, friendly summary by default — e.g. detect the Pydantic-validation shape and collapse to "Run failed: N recommendations had invalid quantities (null proposed_qty)" — and tuck the full trace behind a "Show details" disclosure or the existing `title=` tooltip only. Cap the inline text (e.g. `line-clamp-2`) so one failed run can't dominate the list. Strip the `https://errors.pydantic.dev/...` URL from any user-facing string. Longer term, store a sanitized reason server-side (the project rule already forbids interpolating exception text into 5xx detail — apply the same hygiene to persisted run errors).
- **Acceptance:** A failed FVA run shows a one-line plain-language reason (no `[type=…]`, no `pydantic.dev` URL) by default; the raw trace is available only on explicit expand/hover; a test asserts the rendered failed-row text does not contain `pydantic.dev` and is ≤ 2 lines.

- **Category:** consistency
- **Evidence:** `customerAnalytics.png` + digest 3285-3304 — the tab's first visible text is "Total Demand 23.0M cases ↑ 28.1% MoM"; there is no "<Page Title> — <one-line description>" header. `grep` for `<h1>`/`text-2xl`/title strings in `CustomerAnalyticsTab.tsx` returns nothing — the page renders KPI cards as its top element. Contrast every sibling tab, which all lead with a title + description: Portfolio ("Forecast performance and accuracy analytics across your portfolio…" digest 597-599), Item Analysis ("Sales, forecasts, inventory… for a single DFU" 3005-3006), Explorer ("Browse raw data across all domains…" 3185-3186), Clusters ("Group item-location pairs (SKUs) into clusters…" 3720), Data Quality ("Monitor pipeline health…" 2739-2740). Additionally the sidebar labels this item **"Customer Analytics"** (`AppSidebar.tsx:49`) while the only page-level name a user can see is the sub-panel heading **"Customer Demand Map"** — a label drift on top of the missing header.
- **Impact:** Inconsistent IA and an accessibility gap — the page has no heading landmark, so screen-reader users get no page-context announcement and the visual hierarchy starts mid-content. New users can't tell what the tab is for; the sidebar name and the on-page name don't match. It's the only one of 16 tabs missing this affordance.
- **File:** `frontend/src/tabs/CustomerAnalyticsTab.tsx` (add a header block above `KpiSummaryCards`).
- **Recommendation:** Add a consistent header at the top of the tab — an `<h1>` "Customer Map" (or rename the sidebar item to "Customer Map" so label and title agree) plus a one-line description matching the sibling-tab pattern, e.g. "Where demand, lost sales, and fill rate concentrate by geography, channel, and customer." Reuse whatever shared `TabHeader`/title component the other tabs use.
- **Acceptance:** The Customer Analytics tab renders a top-level `<h1>` whose text matches the sidebar label, followed by a one-line description above the KPI strip; a test asserts `getByRole("heading", {level:1})` is present and equals the nav label.

## U10.3 — Data Quality domain cards render a bold red "2 fail" beside a green "100%" score for domains whose only failures are info-severity (Sku_to_item, Sku_to_location: 0 pass / 2 fail / 2 info / 100%), so the grid contradicts itself at a glance — "100% healthy" next to "2 fail" [P3] [consistency]
- **Category:** consistency
- **Evidence:** digest 2853-2864 — `Sku_to_item` and `Sku_to_location` each render **"100% · 0 pass · 2 fail · 0 warn · 2 info"**. `DataQualityTab.tsx:240-256` renders `{d.failed} fail` in `text-red-600` unconditionally, with the muted "{info_fails} info" chip alongside. When `failed === info_fails` (every failure is info-severity, deliberately excluded from the score by the cycle-8 F7.1/U8.3 fix), the card still shows a bold red "2 fail" next to the green "100%" — the very contradiction the score fix was meant to resolve, now reasserted in the count row. Compare `Forecast_to_sku` / `Sourcing_to_location` (digest 2784-2787, 2876-2879): genuine "0% · 2 fail" with no info chip — correctly alarming.
- **Impact:** Re-introduces the "is this domain broken or not?" ambiguity the severity-aware score was meant to kill. A planner scanning the 20-card grid sees "100%" and "2 fail" on the same card and can't trust either without reading the muted info chip. Low severity (the chip + score disambiguate on inspection) but a cheap polish that completes the cycle-8 fix.
- **File:** `frontend/src/tabs/DataQualityTab.tsx:240-256`.
- **Recommendation:** When `failed === (info_fails ?? 0)` (all failures are info-severity), mute the "fail" count (drop `text-red-600`, render grey or as "(info)") so it no longer reads as an alarming defect beside a 100% score. Equivalently, render `failed - info_fails` genuine fails in red and fold info-only fails entirely into the info chip, so the red count reconciles with the score's denominator.
- **Acceptance:** A domain whose failures are all info-severity renders no alarm-red "fail" count beside its 100% score; a test asserts the fail count cell uses muted (non-red) styling when `failed === info_fails`.

## U10.4 — Across Command Center, Action Feed, and AI Planner the exception list is 100% "CRITICAL" with zero High/Medium/Low — 4,252 critical, 0 high (digest 1161-1167) — so the severity filter chips (All / Critical / High / Medium / Low) and the "High Priority" KPI tile are dead controls that always show 0 [P3] [usability]
- **Category:** usability
- **Evidence:** Action Feed KPIs (digest 1157-1167): "Critical 4,252 · High Priority 0". Command Center header (digest 17-27): "Open Exceptions 6142 · 2465 critical". Every one of the ~50 rendered exception rows is badged `CRITICAL` (digest 41-589, 1175-1413). The tabs expose a severity filter — "All / Critical / High / Medium / Low" (digest 30-33) — and a dedicated "High Priority" tile (`ActionFeedPanel.tsx:81-83`), yet High/Medium/Low are structurally empty: the urgency thresholds collapse the entire population into Critical. So the filter chips for High/Medium/Low and the High-Priority tile are always-zero affordances.
- **Impact:** A degenerate severity distribution makes the triage controls misleading — a planner expecting to step down from Critical → High → Medium finds nothing below Critical, and the "High Priority: 0" tile looks like a bug. It also means "Critical" has lost discriminating power (4,252 items can't all be equally urgent). Lower severity because it stems from the exception-engine thresholds (a data/config issue) more than the UI, but the UI advertises a granularity the data doesn't have.
- **File:** exception severity thresholds in the replenishment-exception engine (`common/engines/…`) feeding `fact_replenishment_exceptions`; UI consumers `frontend/src/tabs/inv-planning/ActionFeedPanel.tsx` + `CommandCenterTab.tsx` severity filter.
- **Recommendation:** Either (a) re-bucket urgency so the population spreads across Critical/High/Medium (e.g. tier by financial impact percentile or days-to-stockout), restoring the filter's meaning; or (b) if the data genuinely has only one severity tier right now, hide the empty High/Medium/Low chips and the High-Priority tile rather than show permanently-zero controls.
- **Acceptance:** The severity filter no longer presents always-empty buckets — either High/Medium/Low are populated by a graded threshold, or empty severity chips/tiles are suppressed; a test asserts no zero-only severity control is rendered when its bucket is empty.

## U10.5 — Inv Planning / Command Center / AI Planner all render the planning date as "Sunday, Jun 14" while every exception row is dated "4/1/2026" (M/D/YYYY) and the heatmap/KPI dates are "2026-04-02" / "Nov 25" — three different date formats on adjacent surfaces, none labeled with a vintage [P3] [consistency]
- **Category:** consistency
- **Evidence:** Inv Planning header "Sunday, Jun 14" (digest 1104) vs exception rows "4/1/2026" (digest 1185, 1197, …, `M/D/YYYY` with no leading zeros) vs Portfolio filter date "2026-04-02" (ISO, digest 601) vs heatmap columns "Nov 25 / Dec 25 / Jan 26 / Feb 26" (digest 707-710) vs Item Analysis FROM/TO "2023-04-01" ISO `<option>`s (digest 3042-3114). Four date renderings — long weekday, `M/D/YYYY`, `YYYY-MM-DD`, abbreviated `Mon YY` — coexist with no shared formatter. (The missing "as of" vintage half is the still-open U9.4.)
- **Impact:** Cross-surface date inconsistency adds low-grade cognitive friction and makes the April-vs-June staleness (U9.4) harder to spot, since the planner is already juggling three formats. A single date-format helper would also let the "as of <date>" vintage line (U9.4) land consistently.
- **File:** scattered — `ActionFeedPanel.tsx` / `CommandCenterTab.tsx` row dates (`M/D/YYYY` via `toLocaleDateString()`), `AggregateAnalysisTab.tsx` filter date, heatmap axis formatter. Consolidate via a shared `frontend/src/lib/formatDate.ts`.
- **Recommendation:** Introduce one date formatter (e.g. `Apr 1, 2026` medium style) and apply it to exception-row dates, filter dates, and KPI dates so the app speaks one date dialect; keep ISO only inside raw-data tables (Explorer) where exactness matters. This is the natural carrier for the U9.4 "as of" vintage line.
- **Acceptance:** Exception-row dates, KPI dates, and filter dates render via a single shared formatter (consistent format app-wide); a test asserts the shared formatter is used by the Action Feed row date cell.
