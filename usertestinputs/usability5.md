## U5.1 — Dead `controlTower` / `aiPlanner` tab branches: initial-load redirects to Command Center but `popstate` does not, so back/forward reaches superseded tabs [P1]
- **Category:** information-architecture
- **Evidence:** `useUrlState.ts:12-20` defines `TAB_REDIRECTS = { aiPlanner: "commandCenter", controlTower: "commandCenter", storyboard: "commandCenter", … }` and `getInitialTab()` applies them (`:25`) — so a fresh load of `?tab=controlTower` or `?tab=aiPlanner` correctly lands on Command Center (confirmed: cycle5 digest renders identical Command Center text under both `?tab=controlTower` (digest 1366-1957) and `?tab=aiPlanner` (digest 2073-2663)). **But** `controlTower`, `aiPlanner`, and `storyboard` are still in `VALID_TABS` (`:3`), and the `popstate` handler (`:44-49`) sets `activeTab` from the URL **without applying `TAB_REDIRECTS`** — so a browser Back/Forward to `?tab=controlTower` sets `activeTab="controlTower"`, which then renders the **dead** `App.tsx:249-252` `<ControlTowerTab>` branch (and `:259-262` `<AIPlannerTab>`). Those tab components (`ControlTowerTab` 844-equivalent Command Center clone, `AIPlannerTab.tsx` 566 lines) are still `lazy()`-imported (`App.tsx:43,45`) and bundled, reachable only via this inconsistency.
- **Impact:** Inconsistent navigation: the same URL renders Command Center on a fresh load but a different (stale, possibly broken) tab after a Back press. ~1400 lines of superseded tab code plus their chunks ship in the bundle for branches the IA has retired. A planner who bookmarks/links `?tab=aiPlanner` gets Command Center; one who navigates there via history gets the old tab — non-deterministic.
- **File:** `frontend/src/hooks/useUrlState.ts:44-49` (popstate skips redirects), `frontend/src/App.tsx:43-47,249-262` (dead branches + lazy imports).
- **Recommendation:** Apply `TAB_REDIRECTS` in the `popstate` handler the same way `getInitialTab` does (resolve `urlTab` through the redirect map before `setActiveTab`). Then delete the now-unreachable `controlTower`/`aiPlanner` branches in `App.tsx` and their `lazy()` imports (and `ControlTowerTab`/`AIPlannerTab` files if nothing else references them), or — if these tabs are intended to live on — remove them from `TAB_REDIRECTS` and restore nav entries. Don't keep both a redirect AND a live render branch for the same tab key.
- **Acceptance:** Navigating to `?tab=controlTower` / `?tab=aiPlanner` via fresh load AND via browser Back/Forward both resolve to the same tab; no code path renders a tab that `TAB_REDIRECTS` retires; `grep -n "ControlTowerTab\|AIPlannerTab" src/App.tsx` returns nothing (or the redirect entries are removed) ; `make ui-test` green.

## U5.2 — ECharts is still shipped and used by 8 Customer-Analytics panels, contradicting the documented "echarts deleted, recharts is the standard" rule [P1]
- **Category:** consistency
- **Evidence:** Project rule (CLAUDE.md / MEMORY.md): *"echarts is dropped — recharts is the standard. `EChartContainer` and `echarts-modular` are deleted; do not reintroduce."* Reality: `package.json:41-42` still lists `"echarts": "^6.0.0"` + `"echarts-for-react": "^3.0.6"`; `src/components/echarts-modular.tsx` and `src/components/EChartContainer.tsx` both still exist; and **8 live panels import them** — `CustomerTreemap.tsx:3`, `CustomerHeatmap.tsx`, `ChannelSunburst.tsx`, `OrderPatterns.tsx`, `CustomerLifecycle.tsx`, `DemandFlowSankey.tsx`, `OosImpactBubble.tsx`, `CustomerItemAffinity.tsx` (grep `echarts-modular` → 8 tab files + EChartContainer + tests). In `customerAnalytics.png` the **Customer Concentration treemap renders nearly blank** (a single small filled rectangle) even though `/customer-analytics/treemap` returns a full tree — `<ReactECharts style={{ height: 360 }} />` (`CustomerTreemap.tsx:85`) sets height but **no width**, so the chart collapses to 0-width until the container settles, a known echarts sizing trap.
- **Impact:** The documentation and the codebase disagree on a core stack decision, so a future contributor following MEMORY will assume these charts are recharts and break them. A heavy second charting library (echarts ^6) ships alongside recharts, inflating the bundle. The treemap's no-width render makes the flagship Customer Concentration panel look empty/broken on first paint.
- **File:** `frontend/package.json:41-42`; `frontend/src/components/echarts-modular.tsx`, `EChartContainer.tsx`; the 8 `src/tabs/customer-analytics/*` panels listed above; `CustomerTreemap.tsx:85` (missing width).
- **Recommendation:** Pick one reality and make code + docs agree. Minimum: (a) give every `<ReactECharts>` an explicit `style={{ width: "100%", height: 360 }}` (fixes the blank treemap immediately), and (b) update MEMORY.md/CLAUDE.md to state echarts-modular is the *intended* engine for these 8 heavy CA panels (treemap/sunburst/sankey/heatmap that recharts can't do well), so the "deleted" claim stops being false. Larger: migrate the recharts-capable ones and drop echarts. Either way, the "echarts deleted" line must not coexist with 8 live importers.
- **Acceptance:** Docs and code agree on the charting engine; the Customer Concentration treemap renders its full tree (not a single rectangle) on first paint; every echarts panel has an explicit width; `grep -rn "echarts-modular" src/tabs` either returns nothing OR the docs explicitly sanction it.

## U5.3 — Customer-Analytics chart panels hardcode inline hex colors and aren't theme-aware (forbidden by the chart-color rule) [P2]
- **Category:** accessibility
- **Evidence:** CLAUDE.md: *"Charts: never accept `theme` as a prop. Read from `useThemeContext()` / `useChartColors()`. Inline hex colors are forbidden in `tabs/` and `components/`."* Yet `CustomerTreemap.tsx:41` (`color: ["#ef4444", "#eab308", "#22c55e"]`), `:58-59` (`borderColor: "#fff"`, `"#ddd"`), `CustomerHeatmap.tsx:120` (`["#ef4444", "#eab308", "#22c55e"]`), and `ChannelSunburst.tsx:27` (`"#ef4444", // red` …) all use literal hex with no `useChartColors()`. None read the theme, so the fill-rate red→amber→green ramp and the `#fff`/`#ddd` borders are baked for light mode — in dark mode (the app has Light/Soft/Dark in the sidebar footer) the white/grey treemap borders and fixed hex ramp won't adapt.
- **Impact:** Charts are visually inconsistent with the rest of the themed UI in Soft/Dark modes (white borders on dark cards, a fixed contrast ramp), and the hardcoded ramp ignores any accessible/color-blind palette the theme might provide. Direct rule violation that will fail a future lint/review gate.
- **File:** `frontend/src/tabs/customer-analytics/CustomerTreemap.tsx:41,58-59`; `CustomerHeatmap.tsx:118-120`; `ChannelSunburst.tsx:27+`.
- **Recommendation:** Route the fill-rate ramp and structural border colors through `useChartColors()` (or a shared `RAG` ramp helper that reads the theme) instead of literals; pull border colors from the theme's card/border tokens. Add the `customer-analytics/` dir to whatever lint check forbids inline hex so this can't regress.
- **Acceptance:** `grep -rn "#[0-9a-fA-F]\{3,6\}" src/tabs/customer-analytics` returns no chart-color literals; the treemap/heatmap/sunburst use theme-derived colors and re-render correctly when the sidebar theme toggles Light↔Dark.

## U5.4 — Store Type filter is a flat ~150+ ungrouped `<option>` list in a 144px `<select>` with no typeahead — unscannable [P2]
- **Category:** information-architecture
- **Evidence:** `customerAnalytics.png` + digest 3329-3605. Even after `normalizeLabelOptions()` dedup, the Store Type facet is a long flat list (digest shows ~150+ surviving entries: `A CHAIN GROCERY`, `Adult Entertainment`, `AIRLINE`, `AIRLINES`, `AIRLINE/SHIP/EX`, … through `Z NIGHT CLUB`) rendered as native `<option>`s in a `w-36` (144px) `<select>` (`CustomerAnalyticsTab.tsx:251-260`). There is no search/typeahead, no grouping (On-Premise vs Off-Premise vs Non-Alcoholic), and lexical dupes survive dedup because they aren't case/whitespace-identical (`AIRLINE` vs `AIRLINES` vs `AIRLINE/SHIP/EX`; `CHAIN GROCERY` vs `Chain Grocery Store`).
- **Impact:** A planner cannot realistically find/select a store type from a 150-row native dropdown in a 144px box — they scroll blindly. The near-synonym entries (`AIRLINE`/`AIRLINES`) mean demand is still smeared across buckets the dedup can't merge. The facet is technically populated but practically unusable.
- **File:** `frontend/src/tabs/CustomerAnalyticsTab.tsx:248-261`; backing data `mv_customer_filter_options`.
- **Recommendation:** Replace the native `<select>` with a searchable combobox (typeahead) for Store Type (and ideally Channel), and/or group options under `<optgroup>` by a coarse on/off-premise bucket. Durable fix: add a canonical `store_type_group` / normalized `store_type` column in the MV so the facet exposes ~10-20 grouped buckets instead of 150 raw labels, with the raw label as a secondary drill-in.
- **Acceptance:** Store Type is searchable (type to filter) or grouped into a small number of `<optgroup>`s; a planner can select "Airlines" without scrolling past 150 rows; near-synonym raw labels are either grouped or documented as intentionally distinct.

## U5.6 — Item Analysis FROM/TO date pickers render two full ~36-row raw ISO `<option>` lists with no TO ≥ FROM validation [P2]
- **Category:** usability
- **Evidence:** digest 2977-3071 — the Item Analysis "FROM" control lists every month `2023-04-01 … 2026-03-01` as raw ISO `<option>`s, and "TO" repeats the identical ~36-row list, plus a separate forecast-horizon list runs to `2028-03-01`. The labels are unformatted ISO strings (`2023-04-01`, not `Apr 2023`), and nothing prevents selecting a TO earlier than FROM.
- **Impact:** Two 36-row identical native dropdowns are tedious to scan, the ISO labels read as machine data rather than human months, and an inverted FROM/TO range silently yields an empty/odd chart with no guidance.
- **File:** `frontend/src/tabs/ItemAnalysisTab.tsx` (and its date-range subpanel) — the FROM/TO `<select>` option rendering.
- **Recommendation:** Format option labels as `Mon YYYY`; when FROM changes, constrain TO's options to ≥ FROM (and vice-versa) so an invalid range can't be picked; consider a single range control instead of two mirrored 36-row selects.
- **Acceptance:** FROM/TO labels read as `Apr 2023`; selecting a TO < FROM is impossible (options are constrained or auto-corrected); the chart never renders from an inverted range.
