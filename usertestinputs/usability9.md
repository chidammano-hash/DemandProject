## NEW

## U9.1 — Unified Action Feed counts AND "Financial Impact at Risk" are computed over only the returned 20-row page, so the headline reads "20 critical · $3.6K at risk" while the same data drives Command Center's "2,465 critical · $246K" — a ~70× understatement of exposure on the planner's morning-triage screen [P1] [consistency]
- **Category:** consistency
- **Evidence:** `invPlanning.png` — the Unified Action Feed KPIs read **Total Actions 20 · Critical 20 · High Priority 0 · Financial Impact at Risk $3.6K**. Live `GET /inv-planning/action-feed` returns `summary: {"total": 20, "critical": 20, "high": 0, "financial_at_risk": 3598.89}` with exactly 20 `actions`. Command Center (`commandCenter.png` + digest 14-27), fed from the SAME exception population, shows **Open Exceptions 6142 · 2465 critical**. Root cause is explicit in code: `inv_planning_insights.py:285` does `actions = actions[:limit]` (limit=20), then `:304-317` builds the summary by counting/`sum()`-ing over the **sliced** list — with a dev comment already admitting it (`:302` "Build summary from the full (unsliced) set would be ideal, but for simplicity we summarise what we return"). So "Total Actions", "Critical", and "$ at risk" all describe one page, not the portfolio. The `$3.6K` is the sum of the top-20 dollar impacts; the true critical exposure is orders of magnitude larger.
- **Impact:** The Action Feed is sold as "your morning starting point" and "Priority-ranked actions … across all inventory planning modules." A planner reads "20 critical, $3.6K at risk", concludes the day is light, and moves on — while 2,465 critical exceptions sit unreviewed. The same screen one scroll up (Command Center) says 2,465 critical / $246K, so the two surfaces flatly contradict each other on identical data. This is the highest-trust-cost defect remaining: the headline number understates real risk by ~70×.
- **File:** `api/routers/inventory/inv_planning_insights.py:283-319` (the `actions[:limit]` slice precedes the summary aggregation) → consumed by `frontend/src/tabs/inv-planning/ActionFeedPanel.tsx:70-88`.
- **Recommendation:** Compute the summary over the FULL candidate set before slicing for display. Concretely: run the count/critical/high/`SUM(financial_impact)` aggregation across all rows fetched from the three source CTEs (or a dedicated `COUNT(*)`/`SUM()` query) and only then `actions = actions[:limit]` for the row list. Then relabel the KPIs so the page-vs-population distinction is explicit, e.g. "Critical 2,465" with the list captioned "Showing top 20 by urgency × impact." Keep `%s` params and `get_conn()` per project rules.
- **Acceptance:** With the live data, the Action Feed "Critical" / "Financial Impact at Risk" KPIs reconcile with Command Center's critical count and dollar exposure (within rounding), and the action list is labeled as a truncated top-N; a backend test asserts `summary.critical` / `summary.financial_at_risk` reflect the full population, not `len(actions)`.

So the planner sees the top-3 actions twice, ~200px apart, with no added information.

If kept, render it only when the feed itself is collapsed/below the fold.

## U9.3 — Customer Map MoM deltas ("↑ 28.1% MoM", "↑ 42.9% MoM") carry no accessible label, no comparison-period text, and no partial-month guard — a current partial month inflates the headline against a full prior month [P2] [accessibility]
- **Category:** accessibility
- **Evidence:** `customerAnalytics.png` + digest 3285-3302 — the KPI strip shows "Total Demand 23.0M cases ↑ 28.1% MoM", "Lost Sales (OOS) 461.0K cases ↑ 42.9% MoM", "Active Customers 32.5K ↑ 3.2% MoM". `KpiSummaryCards.tsx:104-111` `DeltaBadge` renders `{arrow} {Math.abs(delta).toFixed(1)}% MoM` inside a plain `<span>` with **no `title` and no `aria-label`** — a screen reader announces "↑ 28.1% MoM" with no period anchor, and there is no guard for the current month being partial. A +28.1% demand jump and +42.9% OOS jump in one month are the classic signature of comparing a partial current month to a complete prior month (or vice-versa); without an "as of" label the planner can't tell whether demand truly surged or the month is simply mid-stream.
- **Impact:** The first numbers on the Customer Map (the deltas) are both inaccessible (no labeled comparison) and potentially misleading (no partial-period framing). A planner may chase a phantom 43% OOS spike that is a calendar artifact. This sharpens the cycle-6/7 unlabeled-MoM pattern (U6.5 fixed Demand History; U7.4 re-pinned this strip) into a correctness concern, since the magnitudes here are large enough to mislead.
- **File:** `frontend/src/tabs/customer-analytics/KpiSummaryCards.tsx:104-111` (DeltaBadge) — and the `/customer-analytics/kpis` delta computation if the current month is included raw.
- **Recommendation:** Add `title`/`aria-label` to the delta span ("Month-over-month vs <prior month>: +28.1%"), reusing the Demand History MoM aria pattern (U6.5). If the latest month is partial, either exclude it from the MoM base or append a `*` + footnote ("latest month partial — comparison annualized/excluded"), mirroring the >200% low-base footnote already added for Demand History.
- **Acceptance:** Each Customer-Map KPI delta exposes an `aria-label`/`title` naming the comparison period; when the current month is partial the delta is footnoted or computed against comparable spans; a test asserts the delta span carries an accessible label.

## U9.4 — Inv Planning exception/action rows are all dated "4/1/2026" — 2.5 months before the planning date of 6/14/2026 — with no "as of" framing, so the "morning triage" reads as stale without the planner knowing the data's vintage [P3] [usability]
- **Category:** usability
- **Evidence:** `invPlanning.png` + digest 1183-1411 — every Action Feed / Command Center exception row shows **4/1/2026** while the Today's Plan header reads **"Sunday, Jun 14"** (planning date 2026-06-14, confirmed in the env banner). The feed copy promises "your morning starting point" and "Auto-refreshes every 60 seconds", implying freshness, yet the newest exception is from April 1 — the underlying `fact_replenishment_exceptions` snapshot is ~10 weeks old. Nothing on the screen tells the planner the exceptions are an April snapshot, not today's.
- **Impact:** A planner triages what looks like today's exceptions but is acting on April data; the "auto-refreshes every 60s" copy actively implies otherwise. Low severity because it is partly a data-state artifact, but the missing "as of <date>" framing turns a stale snapshot into a silent trust trap.
- **File:** `frontend/src/tabs/inv-planning/ActionFeedPanel.tsx` (feed header) + `CommandCenterTab.tsx` exception list header — add a data-vintage line from the max `exception_date` already returned per row.
- **Recommendation:** Surface the snapshot vintage: "Exceptions as of Apr 1, 2026 (snapshot)" near the feed header when `max(exception_date)` lags the planning date by more than N days, and soften the "auto-refreshes every 60s" copy when the underlying snapshot is stale. Reuse the Command Center amber-banner pattern (already used for stale KPIs) for stale exception snapshots.
- **Acceptance:** When the newest exception date trails the planning date materially, the feed shows an "as of <date>" vintage indicator; a test asserts the indicator appears for an April-dated snapshot under a June planning date.

## U9.5 — Data Quality "Last Value (violations)" column shows "0.00" for passing CRITICAL checks, conflating "zero violations (good)" with a numeric value the eye reads as data; the dual-meaning header makes a healthy column scan as ambiguous [P3] [consistency]
- **Category:** consistency
- **Evidence:** digest 2920-2994 — passing critical rows render `completeness_customer_customer_no  critical … 0.00`, `uniqueness_forecast  critical … 0.00`, `completeness_inventory_loc  critical … 0.00`, etc. The header is "Last Value (violations)" (digest 2891). For these checks `0.00` means **zero violations = PASS**, but a column of `0.00` next to red **CRITICAL** severity badges reads at a glance as "critical metric sitting at zero" rather than "critical check with no violations." Meanwhile genuine fails (the WARNING rows at the top, e.g. `…inventory_snapshot_date_not_future … 796460.00`) put a large number in the same column — so the column mixes "violation count" and "passing zero" with no visual separator.
- **Impact:** The flagship DQ catalog's most data-dense column is semantically overloaded: `0.00` (good) and `796460.00` (bad) sit in one "Last Value (violations)" column, and the reader must cross-reference the Status badge to know which is which. Adds cognitive load on every DQ triage. Low severity — the Status column disambiguates — but a quick win.
- **File:** `frontend/src/tabs/data-quality/CheckCatalogPanel.tsx` (the "Last Value (violations)" cell renderer).
- **Recommendation:** Render `0` violations as a muted "0 ✓" / em-dash (or grey it out) when `last_status === "pass"`, and reserve the bold numeric for non-zero violation counts — so a passing row's value cell visually recedes (mirrors the cycle-6 F6.1 "muted pill when passing" treatment, applied to the value cell). Optionally split into a "Violations" count distinct from a "Last Value" measurement where they differ.
- **Acceptance:** Passing checks with zero violations render a muted/zero-distinct value cell rather than an alarming "0.00" equal in weight to a real violation count; a test asserts a passing row's value cell uses the muted styling.
