## U4.1 ‚Äî Data Quality "Check Catalog" is empty and "Last Run: Never" is false, while the page shows 166 checks / 32 recent issues [P1]
- **Category:** usability
- **Evidence:** `dataQuality.png` + digest 1056-1189. The KPI strip shows **166 Total Checks, 116 Passed, 26 Failed, 6 Warnings**, the Domain Health grid is fully populated, and "Recent Issues (32)" lists checks that ran **"11m ago"** ‚Äî yet the central **"Check Catalog (0) ‚Ä¶ No checks configured yet."** table is empty, and the **"Last Run: Never"** KPI contradicts the 11m-ago issues. Live probes confirm the split: `GET /data-quality/dashboard` ‚Üí full domain payload (166 checks summed), `GET /data-quality/history` ‚Üí rich rows with `run_ts: 2026-06-14T06:54‚Ä¶`, but **`GET /data-quality/checks` ‚Üí `{"checks":[]}`**. The catalog table (`CheckCatalogPanel.tsx`) and the `lastRun` value (`DataQualityTab.tsx:111` reduces over the empty `checkList`) both depend solely on that empty `/checks` endpoint.
- **Impact:** The catalog is the one place a planner browses/sorts/drills into individual checks ‚Äî it is unusable. "Last Run: Never" tells the planner DQ has never executed when it ran 11 minutes ago, undermining trust in the whole page. The numbers on the page actively contradict each other.
- **File:** `frontend/src/tabs/DataQualityTab.tsx:111-114` (lastRun derivation), `frontend/src/tabs/data-quality/CheckCatalogPanel.tsx:76,177`; backend `/data-quality/checks` handler under `api/routers/platform/` (returns empty while `/dashboard` + `/history` are populated).
- **Recommendation:** Fix the `/data-quality/checks` query so the catalog returns the same check set the dashboard aggregates (most likely a JOIN/WHERE that drops every row, or it reads a not-yet-populated config table instead of the run results). Derive **Last Run** from the latest `run_ts` in history (already available), not from the empty catalog. Until the catalog is fixed, "Last Run" must never read "Never" when history has rows.
- **Acceptance:** `GET /data-quality/checks` returns ‚â•1 check; Check Catalog header shows a non-zero count matching the dashboard total; "Last Run" shows the real most-recent run time (‚âà11m ago), never "Never" when history exists.

## U4.3 ‚Äî Demand History series "%" badge is single-month MoM (unbounded, e.g. 520.9%) shown as a series-level trend with no label/tooltip [P2]
- **Category:** usability
- **Evidence:** `demandHistory.png` + digest 300-499. Each series row shows a sparkline, a compact volume (`75.8K`), and a green/red `%` with an up/down arrow ‚Äî but values reach `520.9%` (PINNACLE VODKA), `144.7%`, `130.4%`. Code (`WorkbenchPanel.tsx:160-165`) computes the badge as `((last ‚àí prev)/prev)*100` ‚Äî i.e. the **single most-recent month vs. the prior month**, not a trend. There is **no column header and no tooltip** naming the metric (`:201-209` renders the bare `%`).
- **Impact:** A planner scanning the series rail reads "PINNACLE VODKA ‚Üë520.9%" as a high-growth product, when it is one noisy month-over-month spike on a low-volume SKU. As a series-level summary, last-month MoM is the wrong, unbounded statistic and it is unlabeled, so the planner cannot tell what to sort or act on. (Cycle-3 U3.5 added the color/arrow/sparkline; the metric meaning is still unconveyed.)
- **File:** `frontend/src/tabs/demand-history/WorkbenchPanel.tsx:156-165,201-209`.
- **Recommendation:** (a) Add a `title=`/tooltip on the badge: e.g. "Last month vs. prior month (MoM)". (b) Prefer a more stable summary for a series picker ‚Äî a short-window trend (e.g. last-3-mo avg vs. prior-3-mo) or a YoY change ‚Äî which won't swing to 500%+ on a single sparse month. (c) Cap the *displayed* value (e.g. ">+200%") with the true value in the tooltip so one noisy month doesn't dominate the scan.
- **Acceptance:** The badge has a tooltip naming the metric; a value derived from a single noisy month is either smoothed (windowed) or visibly capped with the raw value in the tooltip; no bare unexplained `>100%` in the rail.
