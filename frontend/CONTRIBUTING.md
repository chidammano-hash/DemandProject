# Frontend Contributing Guide

This document lists the conventions the Supply Chain Command Center UI enforces.
Some are also wired into the ESLint config (`eslint.config.js`); the rest are
intended to be caught in code review. Every rule below is also tested (see the
co-located `__tests__/` directories).

---

## 1. React Query keys MUST come from factories

Every module in `src/api/queries/` that exposes queries also exposes a
`<domain>Keys` factory object. Never write raw string arrays inline.

**Bad**

```ts
useQuery({ queryKey: ["exceptions", params], queryFn: fetchExceptions });
```

**Good**

```ts
import { exceptionKeys, fetchExceptions } from "@/api/queries";

useQuery({
  queryKey: exceptionKeys.list(params),
  queryFn: () => fetchExceptions(params),
});
```

Why: centralizing keys lets us invalidate whole trees (`exceptionKeys.list()`
with no args) and prevents stringly-typed drift. A repo-wide grep of
`queryKey: \[` (literal arrays) should return zero matches inside `src/`.

---

## 2. Theme MUST come from context, NOT props

Do not accept `theme` as a prop on any new component. Read it from the
theme context instead.

**Bad**

```tsx
function MyChart({ theme }: { theme: Theme }) { ... }
```

**Good**

```tsx
import { useThemeContext } from "@/context/ThemeContext";
import { useChartColors } from "@/hooks/useChartColors";

function MyChart() {
  const { theme } = useThemeContext();
  const { trendColors, chartColors } = useChartColors();
  ...
}
```

Exception: `ForecastTrendChart` and a few older components still take `theme`
as a prop for backward compat; new code should not follow that pattern, and
any edit to an older component should migrate it when touched.

---

## 3. Every new component/hook/tab has a co-located test

Mirror the directory you're writing into with `__tests__/`:

- `src/components/Foo.tsx` -> `src/components/__tests__/Foo.test.tsx`
- `src/hooks/useBar.ts`    -> `src/hooks/__tests__/useBar.test.ts`
- `src/tabs/<Tab>.tsx`     -> `src/tabs/__tests__/<Tab>.test.tsx`

Use `TestQueryWrapper` from `src/tabs/__tests__/test-utils.tsx` at the root
of every test tree. Mock API modules with `vi.mock("@/api/queries")`.

---

## 4. Toasts: use the built-in Toaster, not sonner

```ts
import { toast } from "@/components/Toaster";

toast.success("Saved.");
toast.error("Failed.");
```

Do NOT install `sonner`. Our Toaster API is intentionally a subset so a later
swap is a two-line change.

---

## 5. Colors: prefer Okabe-Ito for categorical data

For new charts that encode categorical/series colors, opt in to the
color-blind-safe Okabe-Ito palette via:

```ts
import { OKABE_ITO } from "@/constants/colors";
```

The legacy `TREND_COLORS_BY_THEME` stays in place for existing charts that
already have signed-off visuals.

---

## 6. OpenAPI types

API response shapes live in `src/api/generated/schema.ts`, generated by
`npm run gen:types` (requires the backend on `:8000`). Do not hand-edit
that file; regenerate it whenever the FastAPI schema changes and commit the
diff. Query modules in `src/api/queries/` may import types from there when
convenient, but runtime mocks still live in the query modules themselves.

---

## 7. Running tests locally

From `frontend/`:

```bash
# One-shot
PATH="/opt/homebrew/bin:$PATH" /opt/homebrew/bin/node node_modules/.bin/vitest run --reporter=dot

# Watch mode
PATH="/opt/homebrew/bin:$PATH" /opt/homebrew/bin/node node_modules/.bin/vitest
```

The `make test-all` target runs everything. See `CLAUDE.md` at the repo root
for the full project-wide rules.
